Este documento detalla cada corrección realizada en respuesta al feedback de la cátedra, con la justificación técnica de cada decisión y las respuestas a las 5 preguntas de §5.

§2 — Diseño de APIs

A. Ambigüedad entre PATCH /pedidos/{id} y PATCH /envios/{id}

Observación de la cátedra: En PATCH /pedidos/{id}, el body solo acepta estado (enum: en_transito | entregado | cancelado) según el documento. Sin embargo en PATCH /envios/{id} se acepta tanto estado como track_code. Ambos endpoints son para vendedores y actualizan datos del envío — ¿son realmente dos endpoints distintos o hay solapamiento de responsabilidades? La descripción de /pedidos/{id} decía "Actualizar estado de envío de una orden", lo que generaba ambigüedad.
Corrección realizada (openapi.yaml): Se actualizó el summary de PATCH /pedidos/{id} a "Actualizar estado del pedido — avanzar envío o cancelar (solo vendedores)" y se agregó un campo description detallado que explica las dos funciones del endpoint. También se precisó la descripción del campo estado dentro del requestBody.
Justificación técnica: Los dos endpoints son necesarios y tienen responsabilidades diferenciadas:
Endpoint Entidades que modifica Función principal
PATCH /pedidos/{id} Envio.estado (para en_camino, entregado)
Pago.estado + Carrito.estado + restauración de stock (para cancelado) 		Vista unificada del vendedor: avanzar el estado del despacho o cancelar el pedido completo (con restauración de stock atómica). Es el único endpoint que puede cancelar un pedido.
PATCH /envios/{id} Envio.estado + Envio.track_code Gestión logística fina: carga el código de seguimiento del transportista y avanza entre los estados de despacho (incluye preparando, que no está en /pedidos). No puede cancelar.
Hay solapamiento parcial: ambos pueden actualizar Envio.estado para en_camino y entregado. El solapamiento es intencional — el vendedor puede usar la vista simplificada de pedidos o la vista logística detallada según el flujo en el que esté. La diferencia clave es que /pedidos/{id} es el único que puede cancelar (y el único que tiene cancelado como estado válido), mientras que /envios/{id} es el único que puede actualizar el track_code y tiene el estado preparando.

B. DELETE /carritos con query param en lugar de path param

Observación de la cátedra: El endpoint DELETE /carritos quita un ítem usando id_producto como query param. Semánticamente, DELETE /carritos/{id_producto} (path param) sería más REST-puro. El uso de query param en un DELETE es inusual y puede generar confusión.
Corrección realizada (openapi.yaml): Se agregó un campo description al método DELETE de /carritos explicando la decisión de diseño y por qué se optó por query param.
Justificación técnica: En este sistema, el carrito activo del usuario es el recurso que se modifica — y ese carrito se identifica por el token JWT, no por un ID en la URL. El parámetro id_producto es un selector que indica qué ítem remover del carrito activo del usuario autenticado.

Una URL DELETE /carritos/{id_producto} sería semánticamente ambigua: sugeriría que id_producto identifica un carrito, cuando en realidad identifica un producto dentro del carrito del usuario actual. Con query param queda claro que el recurso a modificar es el carrito activo del usuario autenticado y que id_producto es un filtro sobre su contenido.

Una alternativa REST-pura más correcta sería DELETE /carritos/items/{id_producto}, que expone los ítems como sub-recurso. Se reconoce que esta forma sería más idiomática; la decisión de usar query param se mantuvo por consistencia con la ruta existente POST /carritos que usa el mismo endpoint base para agregar ítems.

§3 — Pipelines de Datos

C. Inconsistencia entre reservacion_minutos: 5 y la implementación real

Observación de la cátedra: La reserva de 5 minutos figura en la respuesta del checkout (reservacion_minutos: 5) y en el documento de APIs, pero el pipeline de compra y el ADR reconocen que esta funcionalidad quedó simplificada. Hay inconsistencia entre lo que la API promete (5 minutos de reserva) y lo que el sistema implementa (sin liberación automática del stock). Debería unificarse la documentación: o eliminar reservacion_minutos del response schema, o documentar explícitamente en el pipeline que la liberación automática no está implementada.
Corrección realizada:
  • openapi.yaml: se eliminó el campo reservacion_minutos del response schema de POST /checkout (respuesta 201).
  • pipeline-datos.md — Pipeline 3, trade-offs: se agregó una nota explícita en la fila "Decremento permanente vs. reserva temporal" explicando que el campo fue eliminado y por qué la liberación automática no se implementó.
Justificación técnica: La liberación automática de stock después de 5 minutos requeriría alguno de estos mecanismos: (a) un cron job externo que expire reservas, o (b) un campo reservado_hasta TIMESTAMP en el modelo de datos que se evalúe al volver a hacer checkout. Ninguno de los dos está implementado, principalmente porque el entorno serverless (Vercel) no puede correr procesos en background persistentes (timer en memoria), y agregar un cron externo aumenta la complejidad de infraestructura sin beneficio proporcional al volumen actual del sistema.

En la práctica, el riesgo de stock "bloqueado" es bajo porque el proveedor de pagos responde el resultado (aprobado/rechazado) vía webhook en segundos, no en minutos. El stock se restaura en ese webhook si el pago es rechazado, o queda decrementado si es aprobado. El campo reservacion_minutos prometía un comportamiento que el sistema no cumple, por eso se eliminó del schema.

D. Pipeline de Compra — ¿el stock queda decrementado si el pago es rechazado?

Observación de la cátedra: El pipeline de compra decrementa el stock antes de confirmar el pago. El Pipeline 2 (webhook) no documentaba explícitamente qué pasa con el stock cuando el pago es rechazado.
Corrección realizada (pipeline-datos.md — Pipeline 2): Se agregó el sub-paso 3e. Restauración de stock al Pipeline 2, describiendo que cuando el webhook indica estado = rechazado, el sistema incrementa Producto.stock por cada ítem del carrito dentro de la misma transacción Prisma que actualiza Pago.estado y Carrito.estado.
Justificación técnica: El comportamiento ya existía (documentado en CU-08, excepción 1.1) pero faltaba como paso explícito en el pipeline. La restauración ocurre dentro de prisma.$transaction, garantizando que o todo se actualiza (Pago + Carrito + Stock) o nada lo hace.

Adicionalmente, cuando un vendedor cancela manualmente un pedido vía PATCH /api/pedidos/{id} con estado cancelado, el mismo mecanismo de restauración de stock se aplica (también dentro de una transacción).

E. Motor de Recomendaciones — ¿por qué está categorizado como pipeline?

Observación de la cátedra: El motor es la lógica de negocio de un endpoint REST: dos queries a la BD, cómputo en memoria y respuesta HTTP. No hay etapas desacopladas ni flujo entre sistemas. La documentación del algoritmo es excelente, pero la categorización como "pipeline" es discutible. ¿Qué criterio usó el grupo para incluirlo en este artefacto y no en el documento de APIs?
Respuesta del grupo: El criterio fue que el motor tiene etapas diferenciadas con responsabilidades distintas, análogas a las etapas de un pipeline ETL aunque síncronas:
  1. Ingesta — parámetros HTTP + validación
  2. Extracción — dos queries a PostgreSQL (producto base + candidatos con stock)
  3. Transformación — tokenización de notas, cálculo de Jaccard por campo, score ponderado, blend con historial
  4. Filtrado y ordenamiento — sort descendente + truncado al límite
  5. Salida — JSON al cliente (sin escritura en BD)
A diferencia de un endpoint típico que solo consulta y serializa datos, el motor aplica un algoritmo de múltiples pasos sobre los datos antes de devolverlos. Se lo incluyó en el artefacto de pipelines porque el algoritmo de Jaccard tiene sustancia arquitectónica propia (con trade-offs documentados: Jaccard vs. embeddings, en tiempo real vs. pre-computado, con y sin historial) que justificaba documentarlo en detalle más allá de lo que permite el openapi.yaml.

Se reconoce que la categorización es discutible: en términos estrictos es la lógica de negocio de un endpoint REST (clasificación "On-Demand Request-Response") y no un pipeline con etapas desacopladas en el tiempo. El documento de APIs describe el contrato del endpoint; el documento de pipelines describe el algoritmo interno.

§4 — Decisiones Arquitectónicas (ADRs)

F. ADR-003 — CU-10 y el Sistema de Proveeduría

Observación de la cátedra: ADR-003 menciona el Sistema de Proveeduría como un sistema externo. Sin embargo, CU-10 no aparece en el documento de APIs: no hay ningún endpoint que reciba un webhook de reposición de stock. Si CU-10 está en el alcance del sistema, falta el endpoint correspondiente en el openapi.yaml. Si quedó fuera del alcance, conviene aclararlo en el ADR-003.
Corrección realizada (docs/adrs.md — ADR-003 Consecuencias): Se reescribió la fila correspondiente a CU-10 para que quede explícito que el flujo es OUTBOUND: el marketplace envía email al proveedor, no al revés. La fila anterior decía "el marketplace envía un email al proveedor y no espera respuesta" pero no aclaraba que el proveedor nunca envía ningún webhook de vuelta.
Justificación técnica: CU-10 (Restock Automático) es un flujo saliente (OUTBOUND):
  1. El checkout decrementa el stock.
  2. El sistema evalúa si el stock resultante cayó por debajo del umbral crítico (5 unidades, constante STOCK_CRITICO en lib/stock.ts).
  3. Si sí, invoca lib/notificaciones.ts de forma fire-and-forget para enviar un email al email_contacto del proveedor (tabla ProveedorProducto).
  4. El proveedor despacha la mercadería. Al recibirla, el vendedor actualiza el stock manualmente desde su panel (CU-06).
El Sistema de Proveeduría no envía ningún webhook entrante al marketplace. Por lo tanto no existe ni se necesita ningún endpoint en el openapi.yaml para CU-10. La ausencia del endpoint no es un gap: refleja correctamente que la comunicación es unidireccional (marketplace → proveedor) y no require un receptor HTTP en el sistema.

G. ADR-004 — Reserva temporal de stock: ¿qué solución se adoptó?

Observación de la cátedra: ADR-004 propone como solución un "campo reservado_hasta TIMESTAMP" o "cron job externo", pero no indica cuál de las dos opciones adoptó el grupo ni si alguna está implementada. Dado que reservacion_minutos: 5 aparece en la respuesta del checkout, es razonable que se aclare en el ADR cómo se resolvió.
Corrección realizada:
  • openapi.yaml: se eliminó reservacion_minutos: 5 del response del checkout (Cambio C).
  • pipeline-datos.md: se agregó nota explicando que el campo fue eliminado y por qué (Cambio E).
Justificación técnica: Ninguna de las dos opciones se implementó. La decisión fue no implementar liberación automática por las siguientes razones:
  • Cron job externo: Vercel (entorno serverless) no puede correr procesos en background persistentes. Un cron externo (GitHub Actions, Railway Scheduler) agrega complejidad de infraestructura no justificada.
  • Campo reservado_hasta: Requeriría una migración del schema, lógica de evaluación en cada operación que toca stock, y aún así necesitaría un proceso que libere expiradas periódicamente.
  • Razón pragmática: la confirmación de pago llega vía webhook en segundos (no en minutos), por lo que el riesgo de stock "bloqueado indefinidamente" es mínimo en la práctica. Si el pago es rechazado, el webhook restaura el stock en la misma transacción. Si el webhook nunca llega (fallo del proveedor), el stock queda decrementado hasta que el vendedor cancele manualmente — comportamiento documentado como gap conocido.

§5 — Respuestas a las Preguntas de la Cátedra

1 PATCH /pedidos/{id} y PATCH /envios/{id}: ¿actualizan entidades distintas? ¿Por qué dos endpoints?

Sí, con matices. PATCH /pedidos/{id} puede actualizar dos entidades según el estado recibido: Envio.estado (para en_camino y entregado) o bien Pago.estado + Carrito.estado + Producto.stock en transacción (para cancelado). PATCH /envios/{id} actualiza solo Envio.estado y Envio.track_code.

Los dos endpoints coexisten porque tienen funciones diferentes que no se solapan completamente: /pedidos/{id} es la vista unificada del vendedor para gestionar el ciclo de vida de un pedido (incluyendo cancelación con restauración de stock); /envios/{id} es la vista logística para registrar datos de despacho (track_code, estado preparando). El solapamiento en [en_camino, entregado] es intencional — el vendedor puede operar desde cualquiera de los dos contextos.

2 Si el pago es rechazado, ¿el stock queda decrementado permanentemente?

No. El webhook de pagos (POST /api/pagos/webhook) maneja el caso estado = rechazado restaurando el stock de cada ítem del carrito (Producto.stock += CarritoProducto.cantidad) dentro de la misma transacción Prisma que actualiza Pago.estado = "rechazado" y Carrito.estado = "cancelado". Esto garantiza que o todo se revierte o nada lo hace. El comportamiento está documentado en CU-08 (excepción 1.1) y se explicitó en el Pipeline 2 como el nuevo paso 3e en esta corrección.

3 ¿Qué pasaría si el catálogo creciera a 10.000 productos?

Con el algoritmo actual (scan lineal de todos los candidatos + Jaccard en memoria), el tiempo de cómputo crece O(n) por request. Para 10.000 productos, la query inicial traería ~10.000 filas con todos sus campos de notas, y el cómputo en memoria podría superar los 500 ms (viola RNF-5 de latencia).

Soluciones en orden de complejidad creciente:

  1. Cache Redis: guardar los scores pre-computados para cada producto; invalidar cuando se actualiza un producto. Resuelve el cómputo repetido sin cambios en el modelo de datos.
  2. Tabla SimilitudProducto: pre-computar la matriz de similitud en un proceso batch (por ej. nightly) y guardar los top-N pares. Cada request hace una sola query a esta tabla en lugar de computar en memoria.
  3. Extensión pgvector de PostgreSQL: convertir las notas a vectores de embeddings y usar un índice HNSW para búsqueda por similitud vectorial. Escala bien a millones de productos.

El diseño actual está acotado al alcance documentado: catálogo de 100–500 productos activos (Pipeline 1 — "Volumen estimado"), donde el cálculo tarda menos de 50 ms.

4 ADR-003 menciona CU-10 (reposición de stock). ¿Está implementado ese endpoint? ¿Dónde?

CU-10 no requiere un endpoint entrante porque el flujo es OUTBOUND: el marketplace le envía un email al proveedor, no al revés. El Sistema de Proveeduría no envía ningún webhook al marketplace.

La implementación está en:

  • lib/notificaciones.ts — función enviarEmail(), que envía el email al proveedor fire-and-forget.
  • lib/stock.ts — la función checkoutAtomico() evalúa el stock post-checkout y, si es menor o igual al umbral crítico, invoca la notificación sin await.

No hay ningún endpoint HTTP receptor para CU-10 porque el proveedor no necesita llamar al marketplace. El ciclo completa cuando el proveedor despacha la mercadería y el vendedor actualiza el stock manualmente (CU-06 vía PATCH /api/inventario/{id}).

5 ¿Cómo afecta la tensión ADR-003 / ADR-004 al email de confirmación de compra si la función serverless termina antes de que se dispare el envío?

Es un riesgo conocido, documentado en ADR-003 (Consecuencias Negativas): "El modelo fire-and-forget no garantiza entrega del email."

En el webhook de pagos (POST /api/pagos/webhook), el email de confirmación de compra se dispara sin await (fire-and-forget) dentro del handler. Si Vercel termina la función serverless antes de que la promesa del email se resuelva, el email se pierde silenciosamente.

Mitigación actual: el handler devuelve la respuesta HTTP 2xx después de disparar el email. En la implementación de Vercel con Node.js, el runtime generalmente drena las microtasks del event loop antes de cerrar la función, por lo que el email suele completarse en la práctica. Sin embargo, esto no está garantizado formalmente.

Solución real: para garantía de entrega se necesitaría uno de estos mecanismos:

  • Guardar el evento "email pendiente" en la BD y procesarlo con un cron job (patrón Outbox).
  • Usar un proveedor de email con reintentos (Resend, SendGrid) que garantice entrega por su propia infraestructura.
  • Encolar el evento en una cola (SQS, RabbitMQ) desde la función serverless y procesarlo con un worker separado.

Estas soluciones están fuera del alcance de la entrega pero son el camino correcto si se necesita alta confiabilidad en las notificaciones.
